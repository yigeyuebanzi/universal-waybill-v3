# 需求理解与假设说明

本文档用于说明 V3 运单全流程管理系统在题目未给出精确数值、边界规则或产品口径时的理解、取舍和默认假设。原则是：核心规则可配置，关键动作可追溯，跨系统数据以 V2 实时接口为准，V3 只保存自身业务数据和必要快照。

## 1. 总体理解

V3 是独立于 V2 的运单全生命周期管理系统，负责仓库扫描品控、物流异常上报、分级审批、审批后的库存与赔付联动。V2 继续作为录单解析和运单主数据来源，V3 不直接连接 V2 数据库，也不在本地伪造运单主数据。

系统中存在两条必须分离但要关联的状态机：

- 工单状态机：处理手工物流异常和自动品控异常的审批、拒绝、重提、超时、执行和关闭。
- 扫描批次状态机：处理扫描记录、品控规则命中、批次暂扣、重复扫描幂等、快速放行和批次解锁。

两者通过 `scan_records.ticket_id` 关联。审批通过后的赔付、库存动作必须和工单状态变更保持一致，并通过 `approval_records`、`compensation_records.approval_record_id`、`inventory_movements.approval_record_id` 保留追溯链路。

## 2. 分级审批金额阈值

默认设定：

- `0 <= amount <= 500`：只需一级审批。
- `amount > 500`：一级审批通过后进入二级审批。
- 品控扫描自动触发的中高风险工单默认直接进入二级审批，可由 `qc_rules.target_approval_level` 调整。

理由：

- 低金额异常更关注处理效率，一级审批即可闭环。
- 超过 500 元后，赔付、补发或追偿带来的财务风险和库存影响明显增加，应增加二级复核。
- 品控异常发生在出库前，直接影响批次锁定、供应商追偿和库存可用量，风险口径比普通手工上报更高。

落地方式：

- 阈值不写死在业务分支中，存储在 `approval_rules.min_amount`、`approval_rules.max_amount`、`approval_rules.target_level`。
- `approval_rules.timeout_hours` 同时保存对应规则下的审批超时口径。
- 后台规则页可新增、编辑、启停审批规则。

边界处理：

- 金额缺失时默认按 `0` 处理，但会保留 V2 快照来源，审批人可在详情中看到金额来源。
- 若 V2 后续提供更精确金额，V3 在关键动作前重新拉取 V2 详情并刷新 `waybill_snapshots.amount`。

## 3. 审批超时时长

默认设定：

- 一级审批超时：24 小时。
- 二级审批超时：24 小时。
- 一级超时后自动升级到二级审批。
- 二级超时后自动驳回为 `auto_rejected`。

理由：

- 24 小时覆盖跨班次交接，也避免工单在普通审批池中过久悬挂。
- 一级审批超时升级二级，比直接驳回更符合“先保障异常有人处理”的原则。
- 二级审批已经是最高人工复核层级，再继续悬挂会放大库存、赔付和客户体验风险，因此自动驳回并要求重新补充材料更清晰。

落地方式：

- 工单的当前超时时点存储在 `exception_tickets.timeout_at`。
- 超时任务由 `POST /api/jobs/tick` 或等效定时触发机制处理。
- 超时动作写入 `approval_records` 和 `audit_logs`，保留系统自动处理痕迹。

边界处理：

- 已关闭、已完成、已快速放行的工单不再参与审批超时任务。
- 若审批人账号禁用，tick 任务先尝试转交同层级可用审批人；无可用审批人时转交管理员或升级处理，避免永久卡死。

## 4. 重提次数上限

默认设定：

- 拒绝后允许重新提交 2 次。
- 第 3 次被拒后自动关闭为 `auto_rejected`。

理由：

- 两次重提足以覆盖材料补充、异常类型修正、金额修正等常见场景。
- 超过两次仍无法通过，通常说明事实材料不足或业务诉求不成立，应关闭工单，避免无限退回造成审批池噪音。

落地方式：

- 当前重提次数存储在 `exception_tickets.resubmit_count`。
- 上限存储在 `exception_tickets.max_resubmit_count`，默认值为 `2`。
- 重新提交接口为 `POST /api/tickets/[id]/resubmit`。

边界处理：

- 已完成、执行中、快速放行、自动驳回的工单不可重提。
- 重提后重新进入待审批状态，并重新计算 `timeout_at`。

## 5. 物流异常类型映射

默认映射：

| 物流异常类型 | 下游动作 | 是否赔付客户 | 库存影响 | 理由 |
|---|---|---:|---|---|
| 丢件 | 赔付客户，必要时重新发货 | 是 | 重新发货时扣减可用库存 | 货物无法交付，客户损失明确 |
| 破损 | 赔付客户，退货入库或重新发货 | 是 | 退回可用或残损库存；补发扣减库存 | 需按破损程度处理客户赔付和库存回收 |
| 客户拒收 | 退货入库，必要时关闭工单 | 否，默认不赔付 | 退回库存增加 | 默认不是物流责任，除非补充责任证明 |
| 超时未签收 | 继续跟进、赔付客户或重新发货 | 视严重度 | 重发时扣减库存 | 超时存在客户体验损失，但需按时长和责任判断 |
| 地址错误 | 修正地址并重新发货 | 否，默认不赔付 | 重发时扣减库存 | 多数属于信息错误，默认不进入理赔 |

品控异常动作映射：

| 品控异常类型 | 下游动作 | 赔付方向 | 库存动作 |
|---|---|---|---|
| 数量不符 | 复点后放行或补齐 | 向供应商追偿 | 调整锁定量或补齐后解锁 |
| 外观破损 | 退供应商 | 向供应商追偿 | 批次出库到退供 |
| 规格不符 | 重采购 | 向供应商追偿 | 原批次作废，生成采购/追偿记录 |
| 标签错误 | 重贴标后放行 | 默认不赔付 | 批次解锁 |
| 批次异常 | 降级处理或整批退供 | 向供应商追偿 | 批次降级、退供或解锁 |

落地方式：

- 赔付方向写入 `compensation_records.direction`，物流异常使用 `pay_customer`，品控异常使用 `claim_supplier`。
- 库存流水通过 `inventory_movements.movement_type` 区分扣减、入库、退供、作废、降级等动作。

边界处理：

- 同一运单同类型未关闭物流异常不允许重复上报。
- 不同类型异常允许并存，例如品控工单未关闭期间又发生物流异常，但执行联动必须分别记录，不能混用赔付方向。

## 6. 角色权限划分

默认角色：

| 角色 | 权限范围 |
|---|---|
| `operator` 操作员 | 扫描录入、手工上报异常、查看自己相关工单 |
| `level1_approver` 一级审批人 | 处理一级审批范围内的工单 |
| `level2_approver` 二级审批人 | 处理二级审批范围内的工单和升级工单 |
| `qc_supervisor` 品控主管 | 处理品控误判快速放行，可查看品控相关工单 |
| `admin` 管理员 | 管理审批规则、品控规则、兜底转交和全局查看 |

强制规则：

- 上报人不能审批自己提交的工单。
- 非当前层级审批人不能越级审批。
- 前端隐藏按钮只是体验优化，后端接口必须做权限校验。
- 品控主管可以和一级/二级审批角色重叠，但快速放行和审批动作仍分别留痕。

理由：

- 自批自核是真实审批系统中最常见的权限漏洞，必须在后端阻断。
- 品控主管的职责是判断扫描误判，不等同于可以绕过所有审批规则。

落地方式：

- 用户角色存储在 `users.role`。
- 工单当前分配审批人存储在 `exception_tickets.assigned_approver_id`。
- 审批动作写入 `approval_records.approver_id`、`approval_records.level`、`approval_records.action`。

## 7. V2 数据同步频率与一致性策略

默认策略：

- 关键写动作实时调用 V2：异常上报前校验运单存在，扫描前校验 SKU 归属于指定运单。
- 详情展示优先使用实时拉取结果；V2 不可用时允许使用本地快照展示，并明确标注同步时间。
- 列表和监控可使用 V3 本地快照和 `integration_logs`，不要求每次列表查询都实时打 V2。
- V2 调用超时时间为 8 秒；网络错误、超时、`5xx` 最多尝试 2 次；`4xx` 业务错误不重试。

理由：

- 上报和扫描会创建 V3 自有业务数据，必须保证运单和 SKU 的真实性。
- 详情和列表是读场景，可以通过快照降级，避免 V2 短时不可用导致 V3 全站不可用。
- 对 `4xx` 不重试可以避免对“不存在运单”“SKU 不归属”等确定性错误反复调用。

落地方式：

- V2 快照存储在 `waybill_snapshots`，包含 `source`、`synced_at`、运单摘要和 SKU 明细。
- 每次 V2 调用写入 `integration_logs`，包含 `request_id`、接口名、入参摘要、状态码、耗时和错误信息。
- V3 调 V2 使用 `x-api-key` 和 `x-request-id`。

边界处理：

- V2 运单金额或 SKU 明细在 V3 工单处理中发生变化时，以关键动作时的实时 V2 返回为准，并刷新快照。
- 若 V2 不可用，上报和扫描失败并给出明确提示；已有工单详情可以展示缓存，但必须标注“使用本地缓存，同步于 XX 时间”。
- 如果 V2 原本没有对外接口，新接口应挂载在 `/api/v1/integration`，保持字段只增不删、旧接口不改、灰度上线。V3 对未知字段容忍，金额统一按 decimal 字符串接收，避免 V2 从 `int` 升级到 `decimal` 时破坏兼容。

## 8. 品控暂扣超时时长

默认设定：

- 品控暂扣超时：2 小时。
- 独立于审批超时，不复用 `approval_rules.timeout_hours`。
- 超时后强制升级到二级审批。

理由：

- 品控暂扣直接锁定库存批次，会产生压仓、缺货和履约延迟成本，应远短于审批 24 小时。
- 2 小时足以覆盖仓内复点、拍照取证、主管初判等常规操作，同时避免半天以上无人处理。
- 升级二级审批比自动放行更安全，因为暂扣来源通常是规则命中或实物异常，不能由系统静默释放库存。

落地方式：

- 品控暂扣超时时点存储在 `exception_tickets.qc_hold_timeout_at`。
- 批次锁定状态体现在 `inventory_items.locked_qty` 和 `inventory_items.status`。
- tick 任务处理品控暂扣超时，升级后仍使用审批超时的 24 小时口径。

边界处理：

- 品控主管快速放行后，批次解锁，工单关闭，不再参与暂扣超时任务。
- 同一 SKU + 批次已有未关闭品控工单时，重复扫描只追加 `scan_records`，不重置 `qc_hold_timeout_at`。

## 9. 品控规则触发阈值

默认规则：

| 规则 | 默认阈值 | 默认严重度 | 默认审批层级 |
|---|---:|---|---:|
| 数量差异 | 超过 5% 或绝对差异大于 2 件 | medium | 2 |
| 外观破损 | 破损等级达到 2 级 | high | 2 |
| 规格偏差 | 偏差超过 3% | medium | 2 |
| 标签错误 | 任意标签错误直接触发 | medium | 2 |
| 批次异常 | 命中风险批次直接触发 | high | 2 |

理由：

- 数量差异和规格偏差允许轻微操作误差，但超过阈值后会影响库存和客户收货准确性。
- 外观破损和批次异常通常涉及供应商质量责任，默认提高风险级别。
- 标签错误虽然可能通过重贴标修复，但会造成错发风险，因此直接触发。

落地方式：

- 规则存储在 `qc_rules.condition_type`、`qc_rules.condition_config`、`qc_rules.severity`、`qc_rules.target_approval_level`。
- 规则命中信息写入 `scan_records.matched_rule_id` 和 `scan_records.rule_snapshot`。
- 后台规则页可新增、编辑、启停、删除规则，不通过改代码调整阈值。

边界处理：

- 多条规则同时命中时，按 `qc_rules.priority` 和严重度选择优先规则。
- 规则被修改后，已生成扫描记录保留当时的 `rule_snapshot`，便于复盘。

## 10. 品控主管权限边界

默认设定：

- 品控主管可以执行“误判快速放行”。
- 快速放行必须填写原因，写入审批记录和审计日志。
- 快速放行只适用于扫描自动触发的品控工单，不适用于手工物流异常工单。
- 品控主管可以兼任审批人，但不能审批自己上报或自己扫描触发并负责处理的工单。

理由：

- 快速放行是为了解决扫描误判和规则误命中的效率问题，不是绕过审批的通用后门。
- 只允许品控主管操作并要求留痕，能在效率和风险控制之间取得平衡。

落地方式：

- 快速放行接口为 `POST /api/fast-release`。
- 后端校验 `qc_supervisor` 或 `admin` 角色。
- 操作写入 `approval_records.action = fast_release`，并更新批次锁定状态和工单状态。

边界处理：

- 未填写放行原因不允许提交。
- 已进入执行中、已完成或已自动驳回的工单不能快速放行。

## 11. 单租户假设

默认设定：

- 当前按单租户、单默认仓实现。
- `users.warehouse_id` 和 `inventory_items.warehouse_id` 保留仓库字段，但不做真实多租户隔离。

理由：

- 当前 V2 数据没有明确暴露租户、商户、仓库权限边界，强行实现多租户会引入虚假的权限假设。
- 题目允许在 V2 数据不区分租户时说明单租户假设。

如果后续支持多租户，需要补充：

- `tenant_id`：用户、运单快照、异常工单、审批记录、库存、赔付、规则表均需增加。
- `warehouse_id`：扫描、库存、工单查询、审批待办均需强制过滤。
- V2 接口也必须返回租户/仓库归属，并在上报和扫描时参与真实性校验。

## 12. 一致性与幂等假设

默认设定：

- 审批状态变更、审批记录、赔付记录、库存流水、批次解锁尽量在同一个数据库事务中完成。
- 重复审批通过/拒绝、后台任务重试、重复扫描都必须具备幂等保护。
- 幂等不依赖前端禁用按钮，必须由后端状态前置校验、版本号或唯一约束保证。

落地方式：

- `exception_tickets.version` 用于审批并发冲突检测。
- `approval_records(ticket_id, idempotency_key)` 唯一约束避免重复审批记录。
- `scan_records.ticket_id` 和批次状态用于判断重复扫描是否只追加记录。
- 赔付和库存流水保留 `approval_record_id`，保证可追溯。

## 14. 想向产品经理确认的问题

1. 审批金额应以 V2 运单金额为准，还是允许上报人填写异常损失金额并以损失金额驱动审批？
2. 二级审批超时自动驳回是否符合业务预期，还是应转交管理员继续兜底？
3. 品控误判快速放行是否需要第二人复核，特别是高价值批次或高风险供应商？
4. 地址错误是否一律不赔付，还是要区分客户填错、客服录错、仓库贴错、承运商转运错误？
5. 超时未签收达到多长时间后才允许赔付，是否按承运商 SLA 或客户等级区分？
6. 供应商追偿是否需要对接财务对账系统，还是当前只记录追偿台账即可？
7. V2 是否计划提供运单金额、SKU 明细、更新时间、仓库/租户归属等稳定字段？
8. V2 异常标记回写是否需要在 V2 运单详情页展示，防止 V2 继续按正常运单处理？
9. 多仓、多租户是否属于近期验收范围，如果是，需要补齐租户模型和权限数据来源。
10. AI 建议是否只作为辅助文本，还是未来希望进入规则推荐或自动分类；若进入自动分类，需要明确人工确认边界。
